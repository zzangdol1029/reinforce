## 먼저: R과 G가 뭔가요?

두 기호의 차이를 이해하면 이 페이지 전체가 쉬워집니다.

| 기호 | 이름 | 의미 | 게임 비유 |
|------|------|------|-----------|
| **R** | Reward (보상) | **그 순간 딱 한 번** 받는 즉각적인 점수 | 코인 하나 먹으면 +1 |
| **G** | Return (수익) | **지금부터 게임 끝날 때까지** 받을 보상들의 총합 | 이 지점에서 끝까지 받을 모든 점수 합계 |

구체적인 예시: A → B → C → 종료, 각 보상이 R₀=1, R₁=2, R₂=3 이면

| 값 | 의미 | 숫자 |
|----|------|------|
| R at B | B에서 받는 **즉각** 보상 | **2** |
| G at B | B**부터** 끝까지 받을 **총** 수익 | 2 + 0.9×3 = **4.7** |

> **R은 "지금 이 순간 받는 점수", G는 "지금부터 게임 끝날 때까지 받을 총 점수"**

---

## 전체 맥락

한 **에피소드** 안에서 **여러 상태** A, B, C를 차례로 방문합니다.  
각 상태마다 return G를 구해야 V(s)를 업데이트할 수 있는데,  
이걸 **어떻게 계산하느냐**에 따라 연산량이 크게 달라집니다.

---

## 에피소드 구조

```
A ──(R₀=1)──► B ──(R₁=2)──► C ──(R₂=3)──► □ 종료
```

| 기호 | 의미 |
|------|------|
| A, B, C | 순서대로 방문하는 **상태** |
| R₀, R₁, R₂ | 각 전이에서 받는 **즉시 보상** (이 예시에서는 1, 2, 3) |
| γ (감마) | **할인율** = 0.9. 미래 보상을 현재 기준으로 조금 할인 |
| □ | 에피소드 **종료** |

- **G_A**: A에서 출발해 끝까지 받는 총 수익
- **G_B**: B에서 출발해 끝까지 받는 총 수익
- **G_C**: C에서 출발해 끝까지 받는 총 수익

---

## 방법 1 — Inefficient (비효율적) ❌

각 상태의 G를 **독립된 공식으로 각각 처음부터** 계산합니다.

$$
\begin{aligned}
G_C &= R_2 = 3 \\
G_B &= R_1 + \gamma R_2 = 2 + 0.9 \times 3 = 4.7 \\
G_A &= R_0 + \gamma R_1 + \gamma^2 R_2 = 1 + 0.9\times2 + 0.81\times3 = 5.23
\end{aligned}
$$

코드로 표현하면:

```python
G_C = R2                            # R2 사용: 1번
G_B = R1 + gamma*R2                 # R2 또 사용: 2번째 등장
G_A = R0 + gamma*R1 + gamma**2*R2   # R2 또또 사용: 3번째 등장!
```

### 문제: R₂가 3번 나옵니다

| G 계산 | R₀ | R₁ | R₂ |
|--------|-----|-----|-----|
| G_C    |     |     | ✓  |
| G_B    |     | ✓  | ✓  |
| G_A    | ✓   | ✓  | ✓  |
| **사용 횟수** | 1번 | 2번 | **3번** |

➡️ 상태가 100개면 R₉₉는 **100번** 반복 계산됩니다. 상태가 많을수록 폭발적으로 증가.

---

## 방법 2 — Efficient (효율적) ✅

### 핵심 아이디어

> **"G_B를 이미 계산해 뒀으니까, G_A 계산할 때 그냥 갖다 쓰자."**

$$
\begin{aligned}
G_C &= R_2 = 3 \\
G_B &= R_1 + \gamma \cdot \underbrace{G_C}_{\text{이미 계산됨}} = 2 + 0.9 \times 3 = 4.7 \\
G_A &= R_0 + \gamma \cdot \underbrace{G_B}_{\text{이미 계산됨}} = 1 + 0.9 \times 4.7 = 5.23
\end{aligned}
$$

일반 공식:

$$\boxed{G = R + \gamma \cdot G_{\text{next}}}$$

코드로 표현하면:

```python
G_C = R2                      # R2 사용: 1번
G_B = R1 + gamma * G_C        # G_C 결과만 가져다 씀 (R2 다시 안 씀)
G_A = R0 + gamma * G_B        # G_B 결과만 가져다 씀 (R1, R2 다시 안 씀)
```

| G 계산 | R₀ | R₁ | R₂ |
|--------|-----|-----|-----|
| G_C    |     |     | ✓ (1번) |
| G_B    |     | ✓ (1번) | ← G_C 재활용, R₂ 직접 안 씀 |
| G_A    | ✓ (1번) | ← G_B 재활용 | ← G_B 재활용 |
| **사용 횟수** | **1번** | **1번** | **1번** |

➡️ 각 R 값이 **딱 1번씩**만 사용됩니다.

---

## 왜 "역방향" 이라고 하나?

두 방법 모두 G_C를 먼저 구하는 건 맞습니다.  
차이는 **"그 결과를 저장해두고 재활용하느냐"** 입니다.

효율적 방법이 작동하려면 **뒤쪽(C) → 앞쪽(A)** 순서로 계산해야 합니다.

```
에피소드 진행 방향:  A → B → C → 종료
계산 방향:         A ← B ← C  (역방향!)
```

1. C를 먼저 계산: `G_C = R₂`
2. B 계산: `G_B = R₁ + γ × G_C` ← C 결과 재활용
3. A 계산: `G_A = R₀ + γ × G_B` ← B 결과 재활용

만약 A부터 계산하려 하면? → "G_B가 뭔지 아직 모른다" → 결국 끝까지 펼쳐야 함 → 비효율적 방법으로 돌아감.

---

## 연산 횟수 비교

| 방법 | 상태 3개 | 상태 100개 |
|------|----------|------------|
| Inefficient | 보상 R을 여러 번 중복 계산 | R₉₉가 100번 등장 |
| **Efficient** | **각 R을 1번씩** | **각 R을 1번씩** |

---

## 실제 코드: `reversed(memory)` 의 의미

`mc_eval.py`의 `eval()` 메서드가 바로 이 방법을 씁니다.

```python
G = 0.0
for (state, action, reward) in reversed(self.memory):
    #                          ↑ 에피소드 끝(C)부터 시작(A)까지 거꾸로 순회
    G = self.gamma * G + reward
    #   ↑ G = γ × (이미 계산된 뒤쪽 G) + 지금 이 R
    #   처음엔 G=0이므로: G_C = γ×0 + R₂ = R₂  ✓
    #   다음:           G_B = γ×G_C + R₁       ✓
    #   그다음:         G_A = γ×G_B + R₀       ✓

    self.V[state] += (G - self.V[state]) / self.cnts[state]
```

`G = γ×G + R` 한 줄이 **`G = R + γ×G_next`** 와 동일합니다.  
역방향으로 돌기 때문에, `G` 변수 하나만으로 뒤쪽 결과가 자동으로 누적됩니다.

## 핵심 요약

| 질문 | 답 |
|------|-----|
| 왜 G_B를 알아야 G_A를 계산하는데 효율적인가? | G_B는 **새로 계산하는 게 아니라 이미 만들어둔 값을 가져다 쓰는 것**이기 때문 |
| Inefficient와 Efficient의 진짜 차이는? | **공식의 모양**이 아니라 **같은 R값을 반복 계산하느냐, 한 번만 계산하느냐** |
| γR이 γG로 바뀌는 이유는? | 앞에서 이미 R들을 누적한 G를 그냥 넣으면 되기 때문. 결과는 같고 연산은 훨씬 적음 |
| 왜 역방향인가? | 뒤 결과를 먼저 구해야 앞에서 재활용할 수 있기 때문 |

> **"전에 계산한 값을 저장해뒀다가 갖다 쓴다"** — 이게 전부입니다. 🎯
